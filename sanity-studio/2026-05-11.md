## 3. `LedgerStatementView.tsx` — full descendant aggregation

Statement view's parent-account aggregation previously fetched only direct children:
```groq
*[_type == "accountCode" && parentCode._ref == $id]{ _id }
```
For 110200 → 110210 (sub-group) → 110211, 110212 (leaves), this returned `[110210]`. Transactions are posted to 110211/110212, so `QUERY_TXN` matched nothing — Statement showed "No posted entries for this period" despite the GL list showing activity.

**Fix:** single GROQ fetches the full chart of accounts, then client-side BFS finds all leaf descendants at any depth:
```ts
client.fetch(`*[_type == "accountCode"]{ _id, "parentId": parentCode._ref, isParent }`)
  .then(allCodes => {
    const childrenOf: Record<string, string[]> = {}
    for (const c of allCodes) if (c.parentId) (childrenOf[c.parentId] ??= []).push(c._id)
    // BFS from accountId, collect nodes with no childrenOf entry → leaf descendants
  })
```
Performance: 3 queries total (chart, BF, txns), independent of tree depth. Real-world chart of accounts is ~50–300 docs → fast.

`childIds` state still drives `QUERY_TXN` via `$accountIds` — only the source changed.

## 9. Design discussion — VAT claimable field on payment schema (no code)

Conceptual conversation on adding `vatClaimable: boolean` to the payment schema:
- **If true**: VAT amount → Input VAT account (Dr asset), expense/asset Dr at net amount
- **If false**: VAT capitalized into expense/asset Dr at gross amount, no Input VAT line

Supporting fields: `vatAmount`, `taxInvoiceNumber`, `taxInvoiceDate` (VAT reporting period may differ from accounting date).

Recommended **payment-level** boolean (not line-level) for v1 — simpler, and almost all real-world tax invoices are uniform within. Mixed-claimability cases can be split into two payments. Required (no default) to force conscious choice. Common non-claimable triggers in Thai tax law: no valid tax invoice received, passenger vehicle purchases, entertainment/gifts.

**Status:** awaiting user decision; no schema/code changes yet.

## 10. Auto-create paired ledger doc when accountCode is saved in TreeView

**The problem found**: New account codes added in the freshly-wired TreeView didn't appear in Financial Statements or General Ledger. Investigation showed both `LedgerListPane` and `FinancialStatementView` source their account list from `*[_type == "ledger"]` — so an `accountCode` without a paired `ledger` document is invisible everywhere downstream.

Verified via the Sanity public API:
```
codes: 92, ledgers: 88, codesWithoutLedger: 4
```

The orphans (519100, 519101, + 2 others) had accountCode docs but no ledger docs. The only existing mechanism to create ledger docs was the manual `scripts/seed-ledgers.mjs` (write-token in script; requires CLI run).

**Fix applied in `tools/AccountCodeTreeView.tsx`** — after each successful `client.create({ _type: 'accountCode', ... })` in the save loop, also create a paired ledger doc:

```ts
try {
  await client.create({
    _type:       'ledger',
    accountCode: { _type: 'reference', _ref: doc._id, _weak: true },
  })
} catch (e: any) {
  ledgerFailures.push({ code: n.code, reason: e?.message ?? 'unknown' })
}
```

Weak reference (won't block accountCode deletion). No BF fields set — schema defaults handle that. **No rollback** if ledger creation fails — the accountCode is kept and a warning toast surfaces affected codes + the recovery command (`scripts/seed-ledgers.mjs`).

Also added `useToast` from `@sanity/ui` for the warning toast.

**Backfill** (one-time, fixes the 4 known orphans):
```
cd vertical-signage/sanity-studio
node scripts/seed-ledgers.mjs
```
Idempotent — only creates ledger docs for accountCodes that lack one. Expected output: `Creating: 4` → `Done.`

**Out of scope (deliberate)**: cascade-delete when an accountCode is deleted. Orphan ledger docs after deletion need manual cleanup.

## 11. Noble-be19 kiosk broken on Netlify — root cause: Sanity CORS allowlist

**Symptom**: Live `noble-be19.netlify.app` shows the kiosk shell (clocks, weather, ticker render) but the main video/image area is black with a broken-image icon. Sister site `mahogany-tower.netlify.app` works fine.

**Investigation path:**

1. **Build data sanity** — all 12 playlist URLs return HTTP 200 with correct content types; baked data structurally fine; `builtAt` on local matches live byte-for-byte after CRLF normalization. Not a data issue.

2. **Deploy timing** — Mahogany live deploy is from May 8 17:48 Bangkok (pre-commit `bd50ec1`). Noble-be19 live deploy is from May 11 09:04 Bangkok (post-`bd50ec1`). Code changes between are mostly cosmetic update-popup CSS.

3. **Hidden difference: Service Worker presence.** The May 8 mahogany deploy was an `index.html`-only push; `/sw.js` returns the SPA-fallback HTML (broken SW that never registers). The May 11 noble-be19 deploy includes a working `sw.js`. **This is the only behavioural difference.**

4. **Root cause: Sanity CDN rejects cross-origin requests from the kiosk domains.**

Without `Origin` header: `200 OK`. With `Origin` header (which a browser SW sends): `403 Forbidden`, body `{"statusCode":403,"message":"CORS Origin not allowed","error":"Forbidden"}`.

The kiosk SW uses `mode: 'cors'` in its fetch (sw.js line 65) to enable cacheable responses + Range slicing. When the CORS preflight fails, the SW falls through to `return new Response('Offline — media not cached', { status: 503 })` (line 85). Browser receives 503 → renders broken state for both `<video>` and `<img>`.

**All 5 kiosk origins are blocked:**

```
HTTP/1.1 403  ←  https://noble-be19.netlify.app
HTTP/1.1 403  ←  https://mahogany-tower.netlify.app
HTTP/1.1 403  ←  https://lumpini-24.netlify.app
HTTP/1.1 403  ←  https://the-room-skv21.netlify.app
HTTP/1.1 403  ←  https://noble-geo.netlify.app
```

**Mahogany is one redeploy away from breaking.** Running `deploy.ps1` against it now would push the May 11 build (which includes the SW) and reproduce the same failure.

**Fix selected: Option A — add the 5 origins to Sanity CORS allowlist** (project `awjj9g8u`, dataset `production`, Sanity Manage → API → CORS Origins, credentials NOT required since SW uses `credentials: 'omit'`).

User is adding the allowlist entries themselves. After CORS is propagated:
- SW unregister via DevTools Application → Service workers → Unregister (or test in incognito to bypass)
- Optional `CACHE_NAME` bump in `sw.js` if existing kiosks need forced SW refresh
