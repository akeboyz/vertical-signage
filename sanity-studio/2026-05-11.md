## 3. `LedgerStatementView.tsx` — full descendant aggregation

Statement view's parent-account aggregation previously fetched only direct children:
```groq
*[_type == "accountCode" && parentCode._ref == $id]{ _id }
```
For 110200 → 110210 (sub-group) → 110211, 110212 (leaves), this returned `[110210]`. Transactions are posted to 110211/110212, so `QUERY_TXN` matched nothing — Statement showed "No posted entries for this period" despite the GL list showing activity.

**Fix:** single GROQ fetches the full chart of accounts, then client-side BFS finds all leaf descendants at any depth:
```ts
client.fetch(`*[_type == "accountCode"]{ _id, "parentId": parentCode._ref, isParent }`)
  .then(allCodes => {
    const childrenOf: Record<string, string[]> = {}
    for (const c of allCodes) if (c.parentId) (childrenOf[c.parentId] ??= []).push(c._id)
    // BFS from accountId, collect nodes with no childrenOf entry → leaf descendants
  })
```
Performance: 3 queries total (chart, BF, txns), independent of tree depth. Real-world chart of accounts is ~50–300 docs → fast.

`childIds` state still drives `QUERY_TXN` via `$accountIds` — only the source changed.

## 5. `AssetRegisterView.tsx` — URL-shareable filters

- `?fy=YYYY` — fiscal year (sets the period range; `?asof=` overrides `to` if both present)
- `?status=active|all|in-storage|installed|under-repair|decommissioned|returned`
- `?account=<accountCodeRef>` — Sanity `_id` (matches existing `ar:accountFilter` localStorage handoff)
- `?asof=YYYY-MM-DD` — as-of date

Custom mixes of statuses (not matching one of the 7 named values) aren't representable in URL per spec — reload of a custom mix falls back to default `active` set. Account ref uses `_id` (ugly but consistent with localStorage handoff); can swap to human code (e.g. "116100") later if shareability matters more.

## 9. Design discussion — VAT claimable field on payment schema (no code)

Conceptual conversation on adding `vatClaimable: boolean` to the payment schema:
- **If true**: VAT amount → Input VAT account (Dr asset), expense/asset Dr at net amount
- **If false**: VAT capitalized into expense/asset Dr at gross amount, no Input VAT line

Supporting fields: `vatAmount`, `taxInvoiceNumber`, `taxInvoiceDate` (VAT reporting period may differ from accounting date).

Recommended **payment-level** boolean (not line-level) for v1 — simpler, and almost all real-world tax invoices are uniform within. Mixed-claimability cases can be split into two payments. Required (no default) to force conscious choice. Common non-claimable triggers in Thai tax law: no valid tax invoice received, passenger vehicle purchases, entertainment/gifts.

**Status:** awaiting user decision; no schema/code changes yet.
